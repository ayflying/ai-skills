# 前端构建陷阱与环境变量处理指南

在现代化单页应用 (SPA) 或 SSR 框架（如 Next.js, Vite, Nuxt）中集成 Casdoor 时，构建阶段的变量丢失是常见的问题。

## 1. 静态固化陷阱
**问题**: 在容器运行期配置了环境变量，但前端页面依然不显示 Casdoor 按钮。
**原因**: 前端项目在 `npm run build` 期间，编译器会将环境变量注入到 JS 代码中。如果在构建镜像时环境变量缺失，相关逻辑会被永久“固化”为禁用状态。

## 2. 工程化解决方案
- **构建参数注入**: 在构建脚本或 Dockerfile 中使用 `ARG` (Build Arguments) 明确声明所需的 SSO 配置变量。
- **强制注入验证**: 构建过程中应打印一条日志，确认环境变量是否已成功写入前端可识别的 `.env.production` 或通过编译插件注入。
- **运行时透传**: 如果框架支持，优先使用运行时可读的配置文件而非纯环境变量注入，以提高镜像的通用性。

## 3. 动态显隐逻辑建议
前端代码不应仅通过“环境变量是否存在”来判断是否显示按钮，建议通过后端提供的一个接口（如 `/api/auth/config`）动态获取当前支持的认证方式，这样可以完全避开构建期的变量依赖问题。
